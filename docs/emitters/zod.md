# `@qninhdt/typespec-zod`

Zod schemas + inferred TypeScript types + form metadata for default
form models and `@tableMixin` schemas.

## When to use it

Use this for frontend / API form validation when you want one source
of truth shared with the backend. The Zod emitter is intentionally
narrow â€” it emits **default form models** and `@tableMixin` schemas,
not your full persistence schema.

## Installation

```sh
pnpm add -D @qninhdt/typespec-zod @qninhdt/typespec-orm @typespec/compiler
pnpm add zod
```

## Runtime expectations

The generated package targets:

- TypeScript 5+ with `module: "ESNext"` (or `"NodeNext"`).
- `zod` v4+.
- ESM only.

Schemas, inferred types, and form metadata are emitted in a single
render pass â€” there's no separate type-generation step.

## Configuration reference

```yaml
emit:
  - "@qninhdt/typespec-zod"

options:
  "@qninhdt/typespec-zod":
    output-dir: "./outputs/zod"
    standalone: true
    library-name: "@acme/forms"
    description: "Generated Zod schemas"
    license: "UNLICENSED"
    include: ["#frontend"]
    auto-include-dependencies: true
    int64-strategy: "string"
    branded-scalars: false
```

| Option                      | Type                               | Default                   | Description                                                                                   |
| --------------------------- | ---------------------------------- | ------------------------- | --------------------------------------------------------------------------------------------- |
| `output-dir`                | string                             | compiler default          | Output directory.                                                                             |
| `standalone`                | boolean                            | `false`                   | Emit `package.json`, `tsconfig.json`, code under `src/`.                                      |
| `library-name`              | string                             | â€”                         | Required when `standalone: true`.                                                             |
| `description`               | string                             | `"Generated Zod schemas"` | `package.json` description.                                                                   |
| `license`                   | string                             | `"UNLICENSED"`            | `package.json` license.                                                                       |
| `include` / `exclude`       | string[]                           | â€”                         | Selectors. `#frontend` is the conventional include.                                           |
| `auto-include-dependencies` | boolean                            | `false`                   | Pull required deps transitively.                                                              |
| `int64-strategy`            | `"bigint" \| "string" \| "number"` | `"string"`                | How `int64` and `uint64` columns render. `"string"` is the safe default for JSON round-trips. |
| `branded-scalars`           | boolean                            | `false`                   | Append `.brand("ScalarName")` to user-defined scalars.                                        |

## Output layout

```
src/
  game_platform/
    forms/
      collaboration/
        CreateInvitationForm.ts
  index.ts                     # root barrel
package.json                   # standalone only
tsconfig.json                  # standalone only
```

The root barrel re-exports every emitted model schema, inferred type,
and meta object so consumers can:

```ts
import { CreateInvitationFormSchema, type CreateInvitationForm } from "@acme/forms";
```

## Schema example

```typespec
@scope("frontend")
@data
model CreateInvitationForm {
  @title("Invitee Email")
  @placeholder("friend@example.com")
  inviteeEmail: Demo.Platform.Accounts.User.email;
}
```

Generates:

```ts
// Code generated by @qninhdt/typespec-orm. DO NOT EDIT.
import { z } from "zod";

export const CreateInvitationFormSchema = z.object({
  inviteeEmail: z.email().max(320).describe("Invitee Email"),
});

export type CreateInvitationForm = z.infer<typeof CreateInvitationFormSchema>;

export const CreateInvitationFormMeta = {
  inviteeEmail: {
    title: "Invitee Email",
    placeholder: "friend@example.com",
    inputType: "email",
  },
} as const;
```

## Generated behavior

### Schemas

- Each form model gets a `<Model>Schema` const.
- Each gets a `<Model>` type via `z.infer<typeof <Model>Schema>`.
- `@title` / `@placeholder` / `@inputType` go into a `<Model>Meta`
  object.

### Lookup types

References like `User.email` reuse the source scalar plus its
constraints. The emitter resolves these at compile time so the
generated Zod chain matches the table schema.

### Branded scalars

When `branded-scalars: true`, user-defined scalars (not library
scalars like `email` or `uuid`) get a `.brand("Name")` chain:

```ts
export const UserIdSchema = z.uuid().brand("UserId");
export type UserId = z.infer<typeof UserIdSchema>;
```

This forces nominal typing â€” you can't accidentally pass an `OrderId`
where a `UserId` is expected.

### `int64-strategy`

| Strategy   | Generated                   | Notes                              |
| ---------- | --------------------------- | ---------------------------------- |
| `"bigint"` | `z.bigint()`                | Native JavaScript bigint.          |
| `"string"` | `z.string().regex(/^\d+$/)` | Default. Safe across JSON / fetch. |
| `"number"` | `z.number().int()`          | Loses precision above 2^53.        |
