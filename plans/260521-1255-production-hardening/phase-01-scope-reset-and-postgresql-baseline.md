# Phase 1: Scope Reset and PostgreSQL Baseline

## Overview

Remove stale scope, make PostgreSQL-only support explicit, and ensure the workspace only builds packages that are actually supported.

## Requirements

- Functional: remove `typespec-gorm` remnants and remove local `@qninhdt/typespec-protobuf` package usage.
- Functional: keep Protobuf examples and CI on upstream `@typespec/protobuf`.
- Functional: remove or reject MySQL/SQLite options in Ent and SQLModel.
- Non-functional: no compatibility aliases for removed packages or local Protobuf emitter APIs.

## Architecture

The supported package set becomes: ORM core, Ent, SQLModel, Zod, and DBML. Database emitters are PostgreSQL-targeted. Protobuf contracts are generated by upstream `@typespec/protobuf`, not a repo-owned emitter.

## Implementation Steps

1. Search for `typespec-gorm`, `@qninhdt/typespec-protobuf`, `@typespec/protobuf`, `mysql`, and `sqlite` references.
2. Remove GORM from docs, scripts, package dependencies, examples, and generated-output references.
3. Remove the local Protobuf workspace package and dependency references.
4. Preserve or restore active Protobuf example compilation through upstream `@typespec/protobuf`.
5. Remove MySQL/SQLite dialect options from emitter schemas, docs, and tests.
6. Use hardcoded PostgreSQL Atlas/dev URLs where relevant.
7. Fix CI output paths so Ent, SQLModel, Zod, DBML, and upstream Protobuf generated examples are all verified under `outputs/file-vault` and `outputs/game-platform`.
8. Regenerate checked-in examples after later phases, not in this phase unless needed to prove path correctness.

## Success Criteria

- [ ] `grep -R "typespec-gorm" .` has no supported-source/docs references.
- [ ] `grep -R "@qninhdt/typespec-protobuf" .` has no active source/config/script/example references.
- [ ] `@typespec/protobuf` remains the active Protobuf emitter dependency.
- [ ] Ent and SQLModel options no longer advertise MySQL/SQLite.
- [ ] CI verifies actual generated output paths.
- [ ] Root README lists upstream Protobuf usage and omits GORM.

## Risk Assessment

- Medium: removing the local Protobuf package may expose example schemas that only compile under the custom prototype. Fix schemas to upstream Protobuf semantics rather than restoring the prototype.
- Low: PostgreSQL-only simplification removes misleading behavior.
